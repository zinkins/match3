# ADR-001: Базовая архитектура игры

## Статус

Принято

## Контекст

Проект является тестовым заданием на позицию `Middle / Senior C# Developer`. Архитектура должна демонстрировать:

- масштабируемость;
- поддерживаемость;
- тестируемость логики `Game Core`;
- отделение игровой симуляции от MonoGame;
- готовность к расширению механик и бонусов.

Игра содержит несколько зависимых фаз обработки хода: выбор, обмен, матчинг, бонусы, гравитация, дозаполнение и каскады. При этом визуальная часть не должна определять правила игры.

## Решение

Выбрана архитектура `Game Core + Game Flow + Presentation` с добавлением:

- `State Machine` для явного управления фазами gameplay flow;
- `Domain Events` для передачи результатов симуляции в визуальный слой.

### Терминология (привычнее для game dev)

Названия слоёв выше — это не “игровая” терминология, а описание ответственности. Эквивалентное разбиение, более привычное в индустрии:

- `Game Core` = `Simulation / Core Gameplay / Rules` (ядро симуляции): состояние, правила, инварианты, поиск матчей, применение бонусов, гравитация, генерация новых фишек — без зависимости от MonoGame/платформы.
- `Game Flow` = `Use Cases / Orchestration` (управление ходом): сценарии “попробовать свайп”, “разрешить каскад”, “закончить ход”, “применить бустер”, “проверить конец уровня”. Определяет порядок вызовов логики `Game Core` и границы транзакций шага.
- `Presentation` = `Client / Frontend / View` (ввод/рендер/анимации): экраны, сцены, UI, эффекты, звук, проигрывание анимаций. Не содержит правил — только отображение и отправка команд.

`MVC / MVP / MVVM` — это, в основном, паттерны организации UI-слоя. Они не заменяют разделение `Game Core/Game Flow/Presentation`, но могут использоваться *внутри* `Presentation` (например, `Screen + Presenter`/`Controller`, который вызывает `Game Flow`).

Дополнительно фиксируются следующие обязательные решения:

- `value objects` для координат и ходов;
- unit tests для слоя `Game Core`;
- отдельный pipeline обработки хода;
- анимации, построенные поверх domain events.

## Последствия

### Плюсы

- логика `Game Core` не зависит от MonoGame и тестируется изолированно;
- игровой flow выражен явно и не размазан по `Update`;
- новые бонусы и механики можно добавлять без переписывания всей системы;
- анимации и UI не нарушают инварианты `Game Core`;
- архитектуру легко объяснить на техническом интервью.

### Минусы

- больше начальной инфраструктуры, чем в простом `screen-driven` подходе;
- выше стоимость первого прототипа;
- требуется дисциплина в разделении слоёв.

## Почему не выбраны альтернативы

- `MVC / MVP / MVVM`: это паттерны UI-слоя, а не архитектура симуляции. Их можно применять внутри `Presentation`, но они не решают задачу отделения правил игры и обработки каскадов от фреймворка рендера.
- `ECS-lite`: расширяемо, но для данного тестового задания выглядит избыточно.
- `Reactive-first`: хорошо подходит для UI, но не должен быть основой доменной симуляции.
- `Monolithic GameScreen`: быстрее в старте, но плохо показывает инженерный уровень.

## Практическое следствие для реализации

- Вся игровая логика размещается в `Game Core` и `Game Flow`.
- `Presentation` только рисует, принимает ввод и проигрывает анимации.
- `MonoGame` используется как render/input framework, а не как место хранения правил игры.

### Минимальный каркас для `Presentation` (MonoGame)

Рекомендуемый “скелет” `Presentation`, который хорошо сочетается с доменной симуляцией и каскадами:

- `Screen/Scene` (например, `GameplayScreen`) отвечает за композицию: `BoardView`, HUD, эффекты, камеру.
- `Presenter/Controller` (например, `GameplayPresenter`) принимает ввод (drag/swipe), превращает его в команду (`TrySwap`) и вызывает сервис `Game Flow`.
- `Game Flow` возвращает результат шага симуляции как набор `Domain Events` (и/или `SimulationStep`), не как “инструкции по анимации”.
- `Presentation` преобразует события в `AnimationQueue` (проигрывание, тайминги, SFX/VFX). По завершении очереди запрашивает следующий шаг у `Game Flow`, пока каскад не закончится.

Так `Update()` в MonoGame остаётся тонким: ввод → команда; события → анимации; окончание анимации → следующий шаг.

### Размещение в solution

- `Match3/Match3.Core/` содержит `Game Core` и `Game Flow` (платформо-независимый код).
- `Match3/Match3.DesktopGL/`, `Match3/Match3.Android/`, `Match3/Match3.iOS/` содержат `Presentation` и платформенные адаптеры (ввод, аудио, файлы, жизненный цикл).

Правило именования:

- в тексте документа используются `Game Core` и `Game Flow`;
- в коде, namespace и путях используются `GameCore` и `GameFlow`.

## Dependency Injection (Composition Root)

### Где находится Composition Root

Composition Root находится в entrypoint-проекте конкретной платформы (в `Presentation`):

- Desktop: `Match3/Match3.DesktopGL/` (в месте создания `Game`/`Game1`).
- Android: `Match3/Match3.Android/`.
- iOS: `Match3/Match3.iOS/`.

`Match3/Match3.Core/` не является composition root и не должен “собирать приложение”: он остаётся платформо-независимым.

При этом `Match3.Core` может предоставлять методы регистрации зависимостей (например, `AddMatch3Core(...)`), но вызывать их должен именно лаунчер (platform entrypoint).

### Какой DI-контейнер допустим

Базовый вариант (рекомендованный для этого проекта): `Microsoft.Extensions.DependencyInjection`:

- лёгкий, кроссплатформенный, привычный в C#;
- хорошо подчёркивает идею “composition root на краю”, без привязки к движку.

Допустимо использовать “Unity-style” контейнеры, если это осознанное решение и описан жизненный цикл:

- `vContainer` — возможен, но его преимущества раскрываются в Unity; в MonoGame интеграцию со scope’ами и жизненным циклом нужно реализовать вручную.
- `Extenject` (Zenject) — также возможен, но ориентирован на Unity (Installer’ы/сцены) и добавляет фреймворк, который в MonoGame не даёт критических преимуществ.

### Правила внедрения зависимостей

- Только `constructor injection`. Никаких статических сервисов и service locator в логике `Game Core` и `Game Flow`.
- `Game Core` не зависит от DI и от MonoGame: в основном это чистые типы, создаваемые через `new`/фабрики.
- `Game Flow` зависит от абстракций (портов) для внешнего мира: `IRng`, `IClock`, `IProgressStore`, `IAudio`, `ILocalization` и т.п. Реализации этих портов предоставляет `Presentation`/платформа.

### Lifetimes (жизненный цикл)

Рекомендованные lifetime’ы для игры:

- `Singleton`: конфиг, локализация, сохранения/хранилище прогресса, аудио, фабрики, stateless-обработчики.
- “Scope на сессию/уровень”: `GameplaySession`/`BoardState`/`GameplayService` (удобно пересоздавать при рестарте уровня и не держать “грязное” состояние в singleton’ах).
- `Transient`: мапперы, небольшие stateless-компоненты, обработчики команд/событий.

Если используется `Microsoft.Extensions.DependencyInjection`, scope уровня создаётся вручную (`IServiceScope`) при старте уровня/матча и уничтожается при выходе/рестарте.
